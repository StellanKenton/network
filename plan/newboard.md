# 4G 模块启动的硬件逻辑说明

## 1. 说明

这份文档只保留后续移植真正需要的信息：

1. 硬件上有哪些控制信号。
2. 每个信号对应哪个 IO 口或扩展 IO 端口。
3. 启动 4G 模块时各信号的电平关系和时序。
4. USB 侧何时算“已经可以开始 AT 通讯”。

这里不再按当前工程的文件、函数、任务名来组织，因为后续移植到其他项目时，那些文件和接口大概率都不存在。

在当前板子上实际对应的是 `TCA9535` 这一类 16bit I/O 扩展芯片。

## 2. 启动 4G 模块所需的硬件分层

当前板级启动逻辑可以抽象为三层：

1. 扩展 IO 芯片层
   - 控制 4G 模块供电使能。
   - 控制 USB 数据通道复用。
2. MCU 本地 GPIO 层
   - 控制 4G 模块 `PWRKEY`。
   - 控制 4G 模块 `RESET`。
   - 读取 4G/5G 版本识别脚。
3. USB Host 层
   - 枚举 4G 模块 USB 设备。
   - 枚举成功后，进入 CDC 或串口类数据通讯。

## 3. 硬件信号总览

### 3.1 扩展 IO 芯片侧信号

#### 3.1.0 TCA9535 板级接口引脚

TCA9535 在当前板子上不是走 MCU 硬件 I2C，而是走模拟 I2C，相关板级引脚如下：

- `SCL -> PD11`
- `SDA -> PD12`
- `INT -> PD13`
- `RESET -> PD7`

其中：

- `PD11/PD12` 用于 bit-bang I2C 通讯。
- `PD13` 是 TCA9535 的中断输入脚，MCU 侧配置为输入。
- `PD7` 是 TCA9535 的复位控制脚，MCU 侧配置为输出。

#### 3.1.0.1 模拟 I2C 引脚配置建议

如果后续移植仍然使用模拟 I2C，建议保持下面配置：

- `SCL(PD11)`：输出、开漏、无内部上下拉、100MHz 速度。
- `SDA(PD12)`：输出、开漏、无内部上下拉、100MHz 速度。

注意点：

- 当前板子的 SDA/SCL 使用开漏输出。
- 当前实现依赖外部上拉，不使用 MCU 内部上下拉。
- 如果新板子没有外部上拉电阻，模拟 I2C 可能无法正常工作。

#### 3.1.0.2 INT 与 RESET 引脚配置建议

当前板子的辅助控制脚配置如下：

- `INT(PD13)`：输入，上拉。
- `RESET(PD7)`：输出，推挽，无上下拉。

默认上电状态：

- `RESET` 初始化后拉高，表示 TCA9535 处于正常工作状态。

如果后续项目不使用 TCA9535 中断，也建议至少把 `INT` 保持为稳定输入态，避免引脚悬空。

当前板上和 4G 启动直接相关的扩展 IO 口如下：

- USB 选择脚 1，对应扩展 IO `P01`。
- USB 选择脚 2，对应扩展 IO `P02`。
- USB 选择脚 3，对应扩展 IO `P04`。
- 4G 供电使能脚，对应扩展 IO `PB0`。

### 3.1.1 TCA9535 端口与位定义

TCA9535 是 16bit 扩展 IO，可分成两个 8bit 端口：

- `P0` 端口：对应 `P0.0` 到 `P0.7`
- `P1` 端口：对应 `P1.0` 到 `P1.7`

当前板子的位编号和端口关系可以按下面理解：

- `P01` 实际对应 `P0.1`
- `P02` 实际对应 `P0.2`
- `P04` 实际对应 `P0.4`
- `PB0` 实际对应 `P1.0`

如果按 16bit 掩码来看：

- `P0.1 = 0x0002`
- `P0.2 = 0x0004`
- `P0.4 = 0x0010`
- `P1.0 = 0x0100`

### 3.1.2 TCA9535 配置寄存器

TCA9535 方向配置使用两个寄存器：

- `Configuration Port0 = 0x06`
- `Configuration Port1 = 0x07`

方向规则是：

- 配置位为 `0` 表示输出
- 配置位为 `1` 表示输入

当前板子的实现里，初始化后把整个 TCA9535 配成输出模式，也就是：

- `Port0 配置值 = 0x00`
- `Port1 配置值 = 0x00`
- 合并写入值 = `0x0000`

这表示 `P0.0 ~ P0.7` 和 `P1.0 ~ P1.7` 全部作为输出使用。

从当前板子的 4G 启动需求看，至少下面几个位必须配置成输出：

- `P0.1`，USB 选择脚 1
- `P0.2`，USB 选择脚 2
- `P0.4`，USB 选择脚 3
- `P1.0`，4G 供电使能

如果后续新项目不需要把全部 16 个扩展 IO 都配置为输出，也至少要保证上面这 4 位是输出。

### 3.1.3 TCA9535 输出寄存器写入规则

TCA9535 输出也分两个端口写入：

- `P0` 负责低 8 位输出
- `P1` 负责高 8 位输出

因此在移植时，建议按下面方式理解：

- USB 路由相关位 `P0.1 / P0.2 / P0.4` 写到 `P0` 输出寄存器
- 4G 供电使能位 `P1.0` 写到 `P1` 输出寄存器

这样后续你即使换成别的驱动写法，也不会把低 8 位和高 8 位混掉。

### 3.2 MCU 本地 GPIO 信号

当前板上和 4G 启动直接相关的 MCU IO 口如下：

- `PA10`，用作 `PWRKEY`。
- `PD4`，用作 `RESET`。
- `PB4`，用作 4G/5G 版本识别输入。

### 3.3 USB Host 硬件口

当前板上 USB Host 初始化时直接配置到的口如下：

- `PB14`
- `PB15`

这两个口被复用到 USB OTG HS 相关功能，用于后续 USB 枚举。

USB 复用相关模式可以直接按电平组合理解：

- 4G 通讯模式
   - `P01 = 0`
   - `P02 = 0`
   - `P04 = 0`
- U 盘导出模式
   - `P01 = 1`
   - `P02 = 0`
   - `P04 = 0`
- 烧录模式
   - `P01 = 0`
   - `P02 = 1`
   - `P04 = 0`

## 4. 系统上电后的默认硬件状态

从硬件逻辑上，系统初始化时应该先完成下面几件事：

1. 初始化扩展 IO 芯片的 I2C 通道。
2. 复位扩展 IO 芯片，确保输出状态可控。
3. 将参与 4G 启动的扩展 IO 口配置为输出。
4. 默认把 USB 路由切到 4G 通讯模式。

对当前板子，如果直接按整片初始化，推荐的 TCA9535 方向配置就是：

- `Configuration Port0 = 0x00`
- `Configuration Port1 = 0x00`

推荐默认状态：

- `P01 = 0`
- `P02 = 0`
- `P04 = 0`

这个默认状态的意义是：系统一上电，USB 数据通路先准备给 4G 模块使用。

注意：

- 此时只是路由和控制口状态已就绪。
- 这不代表 4G 模块已经真正完成上电启动。

## 5. 4G 启动的纯硬件逻辑主流程

从移植角度，4G 模块启动主流程建议固定为下面顺序：

1. 打开 4G 模块电源使能。
2. 把 USB 通道切到 4G 模块。
3. 初始化 USB Host。
4. 配置本地 GPIO：`PWRKEY`、`RESET`、版本识别口。
5. 输出 4G 模块开机脉冲。
6. 等待模块硬件稳定。
7. 等待 USB 枚举成功。
8. 再启动协议层 AT 通讯。

也就是说，这套时序不是“先打 PWRKEY 再准备 USB”，而是：

1. 先供电。
2. 先切 USB 路由。
3. 先把 USB Host 接收端准备好。
4. 再执行模块开机动作。
5. 最后等待枚举完成。

这个顺序在迁移时建议保持不变。

## 6. 扩展 IO 对 4G 供电和 USB 路由的具体操作

### 6.1 打开 4G 电源使能

供电使能的纯逻辑如下：

1. 把扩展 IO 上的 4G 供电使能脚置高。
2. 将该位实际写入扩展 IO 的输出寄存器。

核心含义：

- 4G 模块的 3.8V 供电使能，不是 MCU 直控，而是走扩展 IO 的 `PB0`。
- 该位位于高 8 位端口，也就是 `P1.0`。

### 6.2 切换到 4G 通讯模式

USB 路由切换的纯逻辑如下：

1. 先清掉 `P01/P02/P04` 这三个 USB 选择位。
2. 4G 通讯模式下不再拉高任何一个选择位。
3. 将扩展 IO 的低 8 位输出寄存器写回。
4. 延时 `100ms` 等待硬件稳定。

换句话说，本工程里“USB 切到 4G 模块”的电平组合是：

- `P01 = 0`
- `P02 = 0`
- `P04 = 0`

这部分是后续移植时最关键的板级约束之一。如果新硬件上的复用芯片接线不同，必须重新定义这三个控制位与模式表。

## 7. MCU 本地 GPIO 对 4G 模块的开机时序

### 7.1 GPIO 定义

- `PWRKEY -> PA10`
- `RESET -> PD4`
- `4G/5G 版本识别输入 -> PB4`

### 7.2 GPIO 初始化

GPIO 初始化的目标状态应为：

- 开启 `GPIOA/GPIOB/GPIOD` 时钟。
- `PA10` 配置为推挽输出，用作 `PWRKEY`。
- `PD4` 配置为推挽输出，用作 `RESET`。
- `PB4` 配置为输入，用作 4G/5G 版本识别。

### 7.3 4G 上电时序

4G 上电时序如下：

1. 先完成本地 GPIO 初始化
2. `RESET = 1`
3. `PWRKEY = 0`
4. 延时 `50ms`
5. `PWRKEY = 1`
6. 延时 `30ms`
7. `RESET = 0`
8. 延时 `2000ms`
9. `PWRKEY = 0`

这说明当前板子的 4G 模块开机依赖一个较短的 `PWRKEY` 高脉冲，同时配合 `RESET` 管脚释放。

迁移时必须核对新模块手册里的下面三项：

1. `PWRKEY` 有效电平。
2. `PWRKEY` 最小保持时间。
3. `RESET` 是否需要先拉高再释放，还是完全不需要参与上电。

如果换成其他 4G 模块，这一段最容易需要改。

## 8. USB Host 侧的硬件配置

### 8.1 Host 类型

当前板级逻辑使用的是 USB Host，而不是 USB Device。

从移植角度，需要满足下面几个条件：

- MCU 工作在 USB Host 模式。
- 使用的是 USB OTG HS 控制器。
- 板级硬件要么具备外部 PHY，要么具备与你芯片匹配的内置 PHY 方案。
- USB 供电路径必须满足 Host 侧的供电要求。

### 8.2 BSP 初始化内容

USB Host 底层初始化在硬件上做了这些事：

1. 开启 `GPIOB` 时钟。
2. 配置 `PB14/PB15` 为复用功能。
3. 复用到 USB OTG HS 对应功能。
4. 开启 `OTG_HS` 外设时钟。

USB 中断侧做了：

1. 配置中断组。
2. 使能 `OTG_HS_IRQn`。
3. 抢占优先级 `1`，子优先级 `3`。

### 8.3 VBUS 控制

当前板子的 VBUS 控制逻辑是空实现。

这表示当前板级设计里：

- USB Host 侧并没有在软件里主动拉高/拉低 VBUS 开关。
- 更像是板上已经固定完成了供电路径，或者由其他硬件自动处理。

如果你后续的新项目需要 MCU 主动控制 USB VBUS 开关，就必须自己补齐 VBUS 开关和过流检测逻辑，不能直接照搬当前板子的空逻辑。

## 9. 什么时候认为“4G USB 已经可用”

仅仅完成供电使能和开机脉冲还不够，真正能开始通讯，前提是 USB 已经枚举完成。

从纯逻辑上，“4G USB 已经可用”的判据应该是：

1. USB Host 已识别到 4G 模块。
2. CDC 接口已完成枚举。
3. 接收通道已经打开。
4. 上层状态机已经把链路标记为“用户可用”。

如果你在新项目里要重新实现状态变量，可以直接定义一个布尔条件：

- `modem_usb_ready = true`

只有在这个条件成立后，才允许 AT 层开始发命令。

## 10. LTE 协议层在何时开始发 AT

LTE 协议层启动建议遵循下面顺序：

1. 等待 USB Host 已枚举成功。
2. 再额外等待一段稳定时间，当前板子使用的是 `3000ms`。
3. 然后才开始发初始化 AT 指令。

当前板子的典型 AT 初始化顺序是：

1. `ATE0`
2. `AT+GSN`
3. `AT+CPIN?`
4. `ATI`
5. 其他网络注册与 PDP 配置流程

这说明整个启动过程应该分成两个阶段：

1. 板级硬件启动阶段
   - 电源使能
   - USB 路由切换
   - PWRKEY/RESET 时序
   - USB Host 枚举
2. 模块协议初始化阶段
   - 等待 USB 用户态可用
   - 再发 AT 指令做模块初始化

后续项目最好也保持这种分层，不要把开机脉冲和 AT 初始化直接揉在一起。

## 11. 当前工程里可直接复用的启动顺序

如果你后续在别的项目里要按当前板子的方式启动 4G 模块，推荐按下面顺序实现：

1. 初始化 TCA9535/PCA9535 软件 I2C。
2. 复位扩展 IO 芯片。
3. 将扩展 IO 全部或相关位配置为输出。
4. 先把 USB 复用切到 4G 通讯模式。
5. 拉高 `4G_POWER_EN`，打开 4G 模块供电。
6. 初始化 USB Host + CDC 类。
7. 配置本地 GPIO：`PWRKEY`、`RESET`、版本检测脚。
8. 按当前时序输出开机脉冲：
   - `RESET = 1`
   - `PWRKEY = 0`
   - `50ms`
   - `PWRKEY = 1`
   - `30ms`
   - `RESET = 0`
   - `2000ms`
   - `PWRKEY = 0`
9. 等待 USB Host 枚举成功。
10. 枚举完成后把状态置为“USB 已可用”。
11. 再启动 LTE AT 初始化流程。

## 12. 迁移到新项目时最容易遗漏的点

### 12.1 TCA9535/PCA9535 输出缓存

当前板子的这类实现通常会依赖一个“扩展 IO 输出镜像变量”保存当前输出状态，再统一写回寄存器。

迁移时如果只改单个位、却没有维护这个缓存，很容易把其他控制位误覆盖掉。

### 12.2 USB 路由和供电不是一回事

这里有两套控制：

- `TCA9535_4G_POWER_EN` 负责供电。
- `USB_SEL1/2/3` 负责 USB 路由。

两者缺一不可。只开电不切路由，Host 看不到模块；只切路由不开电，模块不会起来。

### 12.3 VBUS 空实现问题

当前板子 VBUS 软件控制为空。如果新板子用了 USB power switch，需要自己加：

- VBUS 开关 GPIO。
- 过流检测 GPIO。
- VBUS 使能与关闭逻辑的具体实现。

### 12.4 CDC 接收链路要单独核对

从移植角度：

- 发送 AT 命令和等待响应的逻辑可以自己实现。
- USB CDC 的接收启动也可以自己实现。
- 但必须确保“USB 接收回调”最终真的写进“AT 接收缓冲”。

所以迁移时除了启动流程外，还要单独确认：

1. CDC 接收回调最终是否把数据写入了你的 AT 接收缓冲。
2. `UsbSend()` 等待响应使用的环形缓冲区是否真的能收到数据。

## 13. 最小复用结论

如果只提炼本工程启动 4G 模块所必须保留的动作，最小集合就是：

1. 初始化扩展 IO 芯片。
2. 配置扩展 IO 输出方向。
3. 拉高 `PB0`，打开 4G 供电使能。
4. 把 `P01/P02/P04` 切到 4G 通讯模式。
5. 初始化 USB Host + CDC。
6. 输出 4G 开机脉冲。
7. 等待 USB 枚举完成并进入“可用”状态。
8. 再开始 AT 指令初始化。

如果后续你要把这套流程抽成通用驱动，建议把它拆成下面三个接口：

1. `ModemBoard_PowerAndRouteInit()`
2. `ModemBoard_BootPulse()`
3. `ModemUsbHost_StartAndWaitReady()`

这样迁移到新项目时，板级电路差异、USB 栈差异、AT 层差异会比较容易分别替换。
